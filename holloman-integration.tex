\begin{abstract}
We present the integration of Holloman structural fingerprint detection into
two production security platforms: the ClamAV antivirus engine and the Zeek
Network Security Monitor.  Holloman fingerprints---128-bit hashes computed via
Hilbert curve mapping and Lanczos-3 downsampling with AVX2 SIMD---group
malware variants by structural similarity.  In ClamAV, we added a new hash
type (\texttt{CLI\_HASH\_HOLLOMAN}), a signature database format
(\texttt{.hlo}), CVD packaging support, and in-tree holloman5 sources.  In
Zeek, we created a C++ wrapper library (\texttt{zeek::holloman}), registered a
new \texttt{HollomanVal} opaque type, added the \texttt{HOLLOMAN} file
analysis plugin, and deployed production Intel feeds.
Both integrations share a common signature pipeline and 1,000-PE-file
benchmark corpus.  ClamAV benchmarks show 14.2~ms/file (121.4~MB/s) with zero
test suite regressions.  Zeek benchmarks show 19.7~ms/file warm cache
(87.6~MB/s) with 94\% test pass rate.  C++ wrapper overhead is measured at
$<$0.1\% of total latency.
\end{abstract}

% ============================================================================
\section{Introduction}
% ============================================================================

Malware variants evade traditional cryptographic hashes by modifying a single
byte.  Holloman fingerprints address this by capturing \emph{structural
similarity}: files with the same code rearranged produce nearly identical
128-bit fingerprints, while unrelated files diverge.  The algorithm maps a
file onto a Hilbert space-filling curve, producing a 2D image; Lanczos-3
interpolation downsamples it to a fixed 16-byte output.  AVX2 SIMD
acceleration provides $\sim$2.5~GB/s throughput on cached data.

We integrated Holloman into two platforms with fundamentally different
architectures: ClamAV (on-demand file scanner) and Zeek (network traffic
monitor).  This report documents both integrations, their test results, and
benchmarks.

type---MD5, SHA1, SHA256---is an enum value with functions for name
resolution, length, and hex parsing.  We added \texttt{CLI\_HASH\_HOLLOMAN}
as the sixth type, following the same pattern.

\subsection{Signature Format}

\texttt{.hlo} files use the 34-character cluster\_id format:

\begin{lstlisting}
k.45664f7c2d1a4f9900000b0800000000:0:Trojan.Fragtor-VBClone
j.28949c54000d8f7e0000080c3a0e0000:*:Trojan.OtherFamily
\end{lstlisting}

Each line: \texttt{<c>.<32hex>:<size>:<malware\_name>}.  The parser
(\texttt{cli\_loadhlo()}) strictly validates 34-char cluster\_ids with a dot
at position 1.  Signatures are stored via \texttt{hm\_addhash\_bin()} with
\texttt{CLI\_HASH\_HOLLOMAN} to avoid contaminating the MD5 hash table.

\section{Zeek Integration}
% ============================================================================

\subsection{Architecture}

Zeek uses a plugin architecture for file hash computation.  Each hash type
is a subclass of \texttt{file\_analysis::Hash} with an associated
\texttt{HashVal} opaque type.  We added \texttt{HollomanVal} following the
SHA256 pattern and a C++ wrapper library in the \texttt{zeek::holloman}
namespace.

\subsection{C++ Wrapper}

The C++ wrapper provides memory-mapped I/O with pre-allocated output buffers:

\begin{lstlisting}[language=C++]
namespace zeek::holloman {
    using Fingerprint = std::array<uint8_t, 16>;
    bool Init(uint32_t order = 13);
    bool FingerprintFileInto(Fingerprint& out, const char* path);
    bool FingerprintBufferInto(Fingerprint& out,
                               const uint8_t* data, size_t size);
    bool IsAVX2Available();  // inlined
}
\end{lstlisting}

\section{Conclusion}
% ============================================================================

Holloman fingerprint detection has been integrated into both ClamAV and Zeek
as first-class hash types.  The ClamAV integration passes 1,311 tests with
zero regressions and performs at 14.2~ms/file (121.4~MB/s).  The Zeek
integration compiles into the core binary with 94\% test pass rate and
performs at 19.7~ms/file warm cache (87.6~MB/s).  Both share a common
signature pipeline.

The key technical finding across both platforms is that C++/wrapper overhead
is negligible ($<$0.1\% of total latency).  Performance is dominated by I/O
strategy and the AVX2 kernel, not language features.  For in-memory
pipelines (Zeek's network capture, ClamAV's warm-cache scans), holloman
fingerprinting adds approximately 2~ms of CPU time per file.
